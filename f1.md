# F1 — Delete the Standalone Agentic Loop

This is the writeup for what was originally tracked as "F1 · Move `runAgentic()` and HTTP MCP orchestration out of bridge.js to Kotlin." After thinking through the architectural options, we picked **option C — delete the agentic loop entirely** instead of porting it. This document explains why and what it changed.

---

## 1. Why we deleted it instead of porting it

Before F1, the project had **two AI engines**:

1. **claude-code (print mode)** — Anthropic's official CLI v2.1.112, the user-facing engine for the terminal. Talks to tools natively, handles MCP servers, supports its own session/permissions model.
2. **`runAgentic()` in bridge.js** — a home-grown ~700-LOC Anthropic streaming loop we wrote before claude-code was working well. Toggled by `!agentic on`, it intercepted user messages and ran its own tool-calling loop against the same Anthropic API, with its own tool implementations (`bash`, `read_file`, `write_file`, `list_dir`, `web_search`, `device_control`) and its own MCP dispatcher.

The agentic engine was a parallel implementation of everything claude-code already did. Every MCP feature shipped twice ("works in print mode" + "works in agentic mode"). Every tool needed two code paths. The MCP docs had to repeatedly call out the dual mode.

Moving the agentic loop to Kotlin would have preserved that duplication and added IPC overhead (Kotlin → bridge.js for stdio MCP dispatch, since stdio MCP procs are owned by the Node process). The simpler win is to delete the second engine altogether — claude-code is the engine, agentic mode goes away, MCP code shrinks to one path.

The user confirmed in conversation: they don't use `!agentic on`. Most users never even discovered it existed.

## 2. What was deleted

Across three stages, three files, ~830 LOC removed and ~50 added (mostly deprecation shims + comments).

### Stage 1 — bridge.js (−683, +21)

Removed:
- `AGENTIC_FILE`, `UNDO_DIR` constants.
- `AGENTIC_TOOLS` array — the 6 tool defs (bash, read_file, write_file, list_dir, web_search, device_control).
- `executeTool()` — the tool dispatcher.
- `deviceControlTool()` — wrapper around the local DeviceControl HTTP server.
- `callProxyStreaming()` — bespoke streaming Anthropic call (used only by `runAgentic`).
- `AGENTIC_SYSTEM_PROMPT`.
- `runAgentic()` itself.
- `agenticEnabled` module-level var.
- `autoApprove` / `loadAutoApprove()` / `saveAutoApprove()` — install-confirmation state. **Not** the same as `loadApproveList` / `saveApproveList`, which power print-mode permissions and stay.
- `INSTALL_CMD_RE`, `installKeyFromCmd()`.
- `pendingConfirms`, `confirmIdSeq`, `waitForConfirm()` — used only by `executeTool`'s install confirm.
- `!agentic`, `!undo`, `!undo list`, `!confirm:` command handlers.
- `state.agHistory` references.
- The two `runAgentic(...)` invocation sites in `handleInput` (the agentic branch and the image-attach detour).
- `9;agentic:` OSC emissions (both attach-time and `!agentic` toggle).
- `!agentic` / `!undo` lines in `!help`.

Kept (still needed by claude-code print mode):
- `webSearch` + `fetchDdgHtml` — the OpenAI→Anthropic proxy intercepts `WebSearch` tool_calls from claude-code and runs them locally via DuckDuckGo. Comment updated to reflect the new (and only) caller.
- `CONFIRM_FILE` + `loadApproveList` / `saveApproveList` — print-mode permission list (`!perm-always` etc.).
- All MCP infrastructure — claude-code uses it via `patchSettings()` injection (stdio) and the MCP-1 stdio shim (HTTP).

`!agentic` survives as a one-line deprecation shim that prints "use the terminal as normal" instead of doing anything.

### Stage 2 — Kotlin (−150, +14)

Removed:
- `SettingsActivity.refreshPreferenceToggles` — the "Agentic Mode" toggle wiring.
- `SettingsActivity.setupActions` — the `switchAgenticMode` listener.
- `activity_settings.xml` — the toggle row + its divider.
- `AppPreferences.isAgenticMode` / `setAgenticMode` / `KEY_AGENTIC_MODE`.
- `TerminalActivity.onActivityResult` image handling (bitmap decode, base64, thumbnail).
- `TerminalActivity.pickImage` — replaced with a toast that says image attach is temporarily unavailable.
- JS bridge methods `submitMessageWithImage`, `sendConfirm`, `cancelPendingImage`.
- `REQUEST_IMAGE` constant.
- Now-unused imports: `Bitmap`, `MediaStore`, `Base64`, `ByteArrayOutputStream`.

### Stage 3 — terminal/index.html (−139, +12)

Removed:
- `#agentic-badge` element + CSS + click listener + `9;agentic:` OSC handler.
- `#btn-image` button + click listener.
- `#img-preview` + `#img-thumb` preview elements.
- `showConfirmDialog` popup (the install confirmation UI driven by `9;confirm:`).
- `9;confirm:` OSC handler in the dispatch loop.
- `termSetImageReady`, `pendingImageAttached`, `pendingImageDataUrl`, `_hideImgPreview`.
- The dual-branch `submitLine` for "with image" vs "plain text" — collapsed back to plain.
- The "📷" empty-message fallback.

## 3. What this means for users

**Nothing visible breaks** for users who only ever used the normal terminal — claude-code in print mode is unchanged. Same tools, same MCP servers, same permissions, same UI.

**What goes away:**
- The `!agentic`, `!undo`, `!undo list` commands. (`!agentic` still recognised but only prints a deprecation note.)
- The "Agentic Mode" toggle in Settings.
- The agentic-mode amber chip at the top of the terminal.
- The image-attach button in the input bar. **This is the biggest user-visible regression.** Image attach used to route through `runAgentic` to build the multimodal content block for the Anthropic API. With `runAgentic` deleted, there's no consumer for the file. The button now shows a toast.

**What stays:**
- All MCP features (MCP-1 through MCP-9). Print mode uses stdio MCP via settings.json injection and HTTP MCP via the MCP-1 stdio shim. Same as before.
- All Quick Ask / Discussion features — they never touched bridge.js to begin with.
- claude-code print mode terminal — the full flow, including tool permissions, Bash card, sub-agent panel, MCP chip, sys bubbles.
- Web search inside claude-code — the OpenAI→Anthropic proxy still intercepts `WebSearch` tool calls and runs them locally via DuckDuckGo (the helper that powered this in agentic mode survives because claude-code needs it).

## 4. Image attach — what comes next

> **Update (commits af6dce1..aea4b73):** shipped, but not the way this section originally proposed. We picked **proxy-layer injection** instead of routing through claude-code — see invariant 33 in CLAUDE.md and the closed TODO entry. The "JSON stdin to claude-code" plan below was abandoned because `--input-format stream-json` is exactly the kind of new flag combo that historically hangs claude-code v2.1.112 on Android Bionic (same class as `--mcp-config`, `--append-system-prompt`, `dangerouslySkipPermissions`). Proxy injection bypasses claude-code entirely and reuses the existing Anthropic→OpenAI multimodal converter that was already in bridge.js, so all OAI providers + ANTHROPIC_API passthrough keep working with zero spawn-time risk. Subscription mode (OAuth) is the one trade-off — it bypasses our proxy, so image attach is API-key-only.

The original (abandoned) plan for posterity:

The right next step is to route image attach through claude-code instead of through the deleted agentic loop. claude-code accepts images in the conversation message content (the same content-block shape we used to construct manually in `runAgentic`). The change would be:

1. Restore `pickImage()` in `TerminalActivity` (gallery picker + base64 encoding).
2. Send the image as part of the user's next message to bridge — either by writing `pending_image.b64`/`pending_image.mime` (familiar pattern) or via a new JS bridge method.
3. `runMessage` in bridge.js builds the multimodal content block before spawning claude-code (`--print` accepts JSON input for images).
4. Restore the IMAGE button and preview UI in `index.html`.

Estimate: ~150 LOC. Done as a follow-up because the F1 scope was already large enough to merit shipping as-is.

Until that lands, users can still see images through claude-code by passing image paths or URLs in text messages and letting the model read them via the `Read` tool (which works for paths it can access) — though that's narrower than gallery-picker attach.

## 6. Invariants that are no longer relevant

Several CLAUDE.md invariants existed specifically because of the dual-engine setup. They're now safe to remove or simplify:

- **Invariant 5** (PTY mode) — already cleaned up earlier, but mentioned "agentic" several times. Keep the print-mode parts; the agentic references are dead.
- **Invariant 22** (`extractResponseFromPty` for `!pty` subprocess output) — independent of agentic mode, stays.
- **Invariant 31** (Permission prompt wiring) — print-mode only, stays.
- **Invariant 32** (`CONFIRM_FILE` format) — the agentic side (`saveAutoApprove`) is gone; only print-mode `saveApproveList` writes the file now. The "legacy plain-array format for backwards compat" note can stay since old installs may still have the legacy format on disk.
- **Invariant 37** (`handleInput` busy-gate) — the `!confirm:` and `!agentic` pre-gate paths are gone; the gate now treats `!perm-*` + `!clear` as the only pre-gate handlers.
- **Invariant 41** (MCP `enabled` field) — unchanged.
- **Invariant 52** (MCP tool availability by mode) — already updated in MCP-4.

The CLAUDE.md TODO entry for F1 should be marked done with a pointer to this file.

## 7. Test plan (manual)

The bridge.js syntax check passes (`node -e "new Function(...)"`), but the codebase wasn't compiled in this environment — no Android SDK available. Before pushing:

- Build the APK locally (`./gradlew assembleDebug`).
- Confirm Settings doesn't crash on open (the row was deleted but old prefs may still have `agentic_mode=true` — `isAgenticMode` is gone so reads will fail loudly if any caller survived; grep for it again).
- Confirm the terminal opens and a normal claude-code message works.
- Confirm `!agentic`, `!agentic on`, `!agentic off` print the deprecation note and don't break the session.
- Confirm `!mcp`, `!mcp-log`, `!mcp-reload` work (MCP path is unchanged but worth verifying since we touched bridge.js extensively).
- Confirm tapping the (now-missing) IMAGE button area doesn't crash. Actually the button is gone, so there's nothing to tap.
- Confirm rebooting the device + Auto-start service still launches a working terminal (BootReceiver path).

If any of those break, the regression is almost certainly in stage 1 (the big bridge.js deletion) since stages 2 and 3 are pure deletions of unreachable code paths.
